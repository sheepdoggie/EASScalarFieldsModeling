# Technical Design Document: Dual Plaintext and Encrypted Evidence Output

## 1. Purpose

This document defines the dual-output evidence handoff for the enforceable rank-3 modeling framework.

The purpose is to let an operator inspect a plaintext evidence package while also producing an encrypted package for an independent verifier. The two packages are not separately assembled. They are the same canonical signed evidence ZIP, represented in two forms:

```text
1. operator_plaintext_package.zip
2. verifier_encrypted_package.r3enc.json
```

Both are bound by a signed transfer manifest to the same canonical package SHA-256.

## 2. Problem Addressed

A signed evidence package proves framework provenance and file integrity. An encrypted package protects confidentiality in transit. If the operator is allowed to encrypt or not encrypt arbitrarily, encryption is merely optional packaging and does not enforce consistency.

The dual-output system fixes this by requiring the framework to:

```text
1. Build the signed canonical evidence package first.
2. ZIP that package deterministically.
3. Emit that ZIP as the operator plaintext package.
4. Encrypt the exact same ZIP bytes for the verifier.
5. Sign a transfer manifest binding both outputs to the same SHA-256.
```

## 3. Trust Boundary

The dual-output system does not hide results from the operator. The operator receives the plaintext ZIP by design.

It does enforce that:

```text
- the verifier receives the same evidence package the operator saw;
- the operator cannot report one package while sending a different encrypted payload without detection;
- post-packaging modification of the plaintext ZIP invalidates the transfer manifest;
- modification of the encrypted payload invalidates the transfer manifest;
- the decrypted verifier payload must hash to the canonical package SHA-256.
```

## 4. Key Types

The framework now uses two independent key types.

### 4.1 Signing key

Algorithm:

```text
Ed25519
```

Purpose:

```text
Sign EVIDENCE_ENVELOPE.json.
Sign TRANSFER_MANIFEST.json.
```

Generated by:

```bash
python -m rank3_enforced.generate_signing_key <private_key.pem> <public_key.pem>
```

### 4.2 Encryption key

Algorithm:

```text
X25519 + HKDF-SHA256 + ChaCha20Poly1305
```

Purpose:

```text
Encrypt the canonical signed evidence ZIP to the verifier recipient.
```

Generated by:

```bash
python -m rank3_enforced.generate_encryption_key <private_key.pem> <public_key.pem>
```

The signing key and encryption key are intentionally separate.

## 5. Output Structure

A dual-output directory contains:

```text
operator_plaintext_package.zip
verifier_encrypted_package.r3enc.json
TRANSFER_MANIFEST.json
TRANSFER_MANIFEST.sig
TRANSFER_PUBLIC_KEY.pem
TRANSFER_SHA256SUMS.csv
```

The plaintext ZIP is the canonical signed evidence package. The encrypted payload is produced by encrypting those exact ZIP bytes.

## 6. Transfer Manifest

The transfer manifest includes:

```text
schema
transfer_mode
canonical_package_name
plaintext_package_name
encrypted_package_name
canonical_package_sha256
plaintext_package_sha256
encrypted_package_sha256
evidence_envelope_hash
certificate_public_key_sha256
recipient_public_key_sha256
encryption_algorithm
signature_algorithm
plaintext_and_encrypted_payload_are_same_canonical_package
timestamp_utc
```

The manifest is signed with Ed25519. Its signature is stored in:

```text
TRANSFER_MANIFEST.sig
```

## 7. Verification

Verification without decryption checks:

```text
TRANSFER_MANIFEST.sig is valid.
operator_plaintext_package.zip hash equals plaintext_package_sha256.
operator_plaintext_package.zip hash equals canonical_package_sha256.
verifier_encrypted_package.r3enc.json hash equals encrypted_package_sha256.
```

Verification with the recipient private encryption key additionally checks:

```text
decrypt(verifier_encrypted_package.r3enc.json) hashes to canonical_package_sha256.
the decrypted ZIP contains a valid framework certificate.
```

Command:

```bash
python -m rank3_enforced.verify_dual_package <transfer_dir> <recipient_private_key.pem>
```

## 8. Execution Command

A dual-output run is created with:

```bash
python run_dual_declarative_overlay.py \
    <overlay.json> \
    <output_dir> \
    <signing_private_key.pem> \
    <recipient_encryption_public_key.pem>
```

## 9. Acceptance Rule

A dual-output package is acceptable only if:

```text
1. TRANSFER_MANIFEST.sig verifies.
2. operator_plaintext_package.zip hashes to canonical_package_sha256.
3. verifier_encrypted_package.r3enc.json hashes to encrypted_package_sha256.
4. Decrypted verifier payload hashes to canonical_package_sha256.
5. The decrypted canonical evidence package has a valid framework certificate.
```

## 10. Limitations

This system does not prevent the operator from seeing or reporting the results. The plaintext package exists for operator analysis.

It guarantees package identity between the operator-visible evidence and verifier-encrypted evidence.
